# Writing to the iterion board

Iterion runtime registers an internal MCP server (`__mcp-board`) for any
agent/judge node whose `capabilities:` list contains a `board.*` entry.
The server exposes these tools, each gated by a single capability:

| Tool FQN | Capability | Purpose |
|---|---|---|
| `mcp__iterion_board__create_issue` | `board.create` | Create a new issue. |
| `mcp__iterion_board__transition_issue` | `board.move` | Move an issue between states. |
| `mcp__iterion_board__set_bot` | `board.assign` | Choose which bot runs the issue — the canonical dispatcher selector. |
| `mcp__iterion_board__assign_issue` | `board.assign` | Set the human/team owner. NOT for bot selection (use set_bot). |
| `mcp__iterion_board__set_labels` | `board.label` | Replace the issue's label list. |
| `mcp__iterion_board__close_issue` | `board.close` | Move the issue to a terminal state. |
| `mcp__iterion_board__list_issues` | `board.read` | List issues (optionally filtered). |
| `mcp__iterion_board__get_issue` | `board.read` | Fetch one issue by ID or short prefix. |

A node only sees the tools its capabilities unlock — anything else is
absent from `tools/list` and rejected with a `capability denied` error
if called directly. (Evoly's emit node grants board.read, board.create,
board.assign, board.label — it does not close issues.)

## Tool input shapes

### `create_issue` (board.create)

```json
{
  "title": "Refactor X (required)",
  "body": "Markdown body (rationale + acceptance criteria).",
  "state": "backlog",
  "labels": ["horizon:next", "source:evolve"],
  "priority": 5,
  "assignee": "alice",
  "blockers": ["native:abcd1234"],
  "fields": {"bot_args": "key=value"}
}
```

`assignee` here is the human/team owner. To choose which **bot** runs
the issue, create it then call `set_bot` (create_issue has no `bot`
field). Setting `assignee` to a bot name still works as a routing
fallback, but `set_bot` is the canonical, unambiguous selector.

`fields` is the issue's freeform metadata map. The default board
registers a `bot_args` text field; you may set it here as best-effort
per-ticket `--var` overrides, but the board MCP create cannot set typed
`bot_args` that the dispatcher reads as a map today. The reliable
dispatch channel is `set_bot` + a self-contained body (the target bot's
dispatch_vars render title + body). For Evoly's emit, leave issues in
`backlog` (do not move to ready).

Returns the created `Issue` object. `id` is autogenerated as
`native:<uuid>` — capture it for downstream calls.

### `transition_issue` (board.move)

```json
{"id": "native:abcd...", "to": "ready"}
```

`id` accepts unambiguous prefixes (first 8 chars of the UUID work).
Fails with `unknown state` if `to` is not declared on the board.
(Evoly does NOT move tickets to ready — promotion is the operator's or
Nexie's decision. Evoly is not granted board.move.)

### `set_bot` (board.assign)

```json
{"id": "native:abcd...", "bot": "feature-dev"}
```

Sets the issue's **bot** — the canonical way to choose which workflow
the dispatcher runs. The dispatcher routes by `bot` first, falling back
to `assignee` only when `bot` is empty. A bot name that resolves to a
file but isn't in the dispatcher's `assignee_workflows` is **skipped
with a warning** rather than silently running the default workflow, so a
typo'd bot fails loudly. Names are matched tolerantly (`feature_dev` ≡
`feature-dev` ≡ `Feature Dev`). Empty string clears it.

### `set_labels` (board.label)

```json
{"id": "native:abcd...", "labels": ["source:evolve", "kind:evolution", "horizon:next"]}
```

Replaces — does not merge. Pass an empty array to clear.

### `list_issues` (board.read) + `get_issue` (board.read)

```json
{"state": "backlog", "label": "source:evolve"}
```

All filter fields are optional. `get_issue` takes `{"id": "..."}`.

## Kanban states

The default board ships these states:

| Name | Terminal | Eligible | Notes |
|---|---|---|---|
| `inbox` | no | no | Bot-emitted findings land here. Operator triages → backlog (promote) or deletes (dismiss). |
| `backlog` | no | no | Initial state — drafts not yet ready. **Evoly's evolution tickets land here.** |
| `ready` | no | **yes** | The dispatcher polls this and dispatches. The operator drags a backlog ticket here to launch it. |
| `in_progress` | no | yes | Dispatcher flips an issue here on claim. |
| `review` | no | no | Bot done, waiting on human review. |
| `done` | **yes** | no | Successful terminal state. |
| `blocked` | **yes** | no | Failure terminal state. |

## The "backlog → operator drags to ready → dispatcher dispatch" pattern

Evoly creates evolution tickets in `backlog` with `set_bot` + a self-
contained body. It does NOT promote them. When the operator (or Nexie)
moves a ticket to `ready`, the dispatcher's polling loop picks it up and
runs the named bot with title + body rendered through that bot's
dispatch_vars.

```
create_issue(state="backlog") + set_bot(bot) ──┐
                                                ├─→ operator/Nexie drags to "ready"
                                                └─→ dispatcher poll → dispatch to bot
```

If you ALSO want an evolution to surface in Nexie's survey as a finding
(not just a backlog card), `create_issue` with `state: "inbox"` plus a
`findings` label is the canonical shape — Evoly's handoff node uses this
for the single "vision ready" summary card.

## Common error cases

- **`capability denied`** (`-32601`): the node was not granted the
  required `board.*` capability. Add it to the `capabilities:` list on
  the agent/judge declaration.
- **`unknown state`**: target state isn't declared on the board. Use
  `list_issues` to discover the board, or run
  `iterion issue board show` from a shell.
- **`required field <name> missing`**: the board declares a required
  custom field that wasn't provided. Add it to `create_issue.fields`.

## Sandbox runs (containerised bots)

When the workflow runs with `sandbox:` active, the host's iterion
binary isn't reachable inside the container. The runtime falls back
to an HTTP transport (`/api/v1/mcp/board` on the iterion server),
authenticated via an ephemeral `X-Iterion-Run` token. Both paths
expose the same tool set and the same error semantics. (Evoly does not
sandbox, so it uses the native stdio transport.)

## Quick sanity check

After your workflow run finishes, you can inspect the resulting
board from a shell:

```bash
iterion issue list --store-dir <workspace>/.iterion/dispatcher
iterion issue board show --store-dir <workspace>/.iterion/dispatcher
```

Or via the studio at `/board` if `iterion studio` is running on the
same workspace.
