# Writing to the iterion board

Iterion runtime registers an internal MCP server (`__mcp-board`) for any
agent/judge node whose `capabilities:` list contains a `board.*` entry.
The server exposes seven tools, each gated by a single capability:

| Tool FQN | Capability | Purpose |
|---|---|---|
| `mcp__iterion_board__create_issue` | `board.create` | Create a new issue. |
| `mcp__iterion_board__transition_issue` | `board.move` | Move an issue between states. |
| `mcp__iterion_board__assign_issue` | `board.assign` | Change the issue's assignee. |
| `mcp__iterion_board__set_labels` | `board.label` | Replace the issue's label list. |
| `mcp__iterion_board__close_issue` | `board.close` | Move the issue to a terminal state. |
| `mcp__iterion_board__list_issues` | `board.read` | List issues (optionally filtered). |
| `mcp__iterion_board__get_issue` | `board.read` | Fetch one issue by ID or short prefix. |

A node only sees the tools its capabilities unlock — anything else is
absent from `tools/list` and rejected with a `capability denied` error
if called directly.

## Tool input shapes

### `create_issue` (board.create)

```json
{
  "title": "Refactor X (required)",
  "body": "Markdown body (rationale + acceptance criteria).",
  "state": "ready",
  "labels": ["horizon:short-term", "source:whats-next"],
  "priority": 5,
  "assignee": "vibe_feature_dev",
  "blockers": ["native:abcd1234"],
  "fields": {"bot_args": "--var,feature_prompt=Add CSV export"}
}
```

Returns the created `Issue` object. `id` is autogenerated as
`native:<uuid>` — capture it for downstream calls.

### `transition_issue` (board.move)

```json
{"id": "native:abcd...", "to": "ready"}
```

`id` accepts unambiguous prefixes (first 8 chars of the UUID work).
Fails with `unknown state` if `to` is not declared on the board.

### `assign_issue` (board.assign)

```json
{"id": "native:abcd...", "assignee": "vibe_feature_dev"}
```

Empty string clears the assignee.

### `set_labels` (board.label)

```json
{"id": "native:abcd...", "labels": ["chore", "blocked"]}
```

Replaces — does not merge. Pass an empty array to clear.

### `close_issue` (board.close)

```json
{"id": "native:abcd...", "to": "done"}
```

`to` is optional; when omitted, defaults to the board's first terminal
state. Rejects non-terminal targets.

### `list_issues` (board.read) + `get_issue` (board.read)

```json
{"state": "ready", "label": "horizon:next-action", "assignee": "vibe_feature_dev"}
```

All filter fields are optional. `get_issue` takes `{"id": "..."}`.

## Kanban states

The default board ships these states:

| Name | Terminal | Eligible | Notes |
|---|---|---|---|
| `backlog` | no | no | Initial state — drafts not yet ready. |
| `ready` | no | **yes** | The dispatcher polls this and dispatches. |
| `in_progress` | no | yes | Dispatcher flips an issue here on claim. |
| `review` | no | no | Bot done, waiting on human review. |
| `done` | **yes** | no | Successful terminal state. |
| `blocked` | **yes** | no | Failure terminal state. |

A workflow can customise the board (different names/colors) via
`iterion issue board init`; the labels above are conventions, not
DSL-enforced names.

## The "move to eligible → dispatcher dispatch" pattern

Whenever you create an issue you want the dispatcher to pick up
immediately, set `state: "ready"` in the create_issue call (or call
`transition_issue` right after). The dispatcher's polling loop walks
the board and dispatches issues whose state has `eligible: true`. If
you leave the issue in `backlog`, it stays as a draft until a human
or another bot moves it forward.

```
create_issue(..., state="ready") ──┐
                                    ├─→ dispatcher poll → dispatch to bot
transition_issue(id, "ready")    ──┘
```

## Common error cases

- **`capability denied`** (`-32601`): the node was not granted the
  required `board.*` capability. Add it to the `capabilities:` list on
  the agent/judge declaration.
- **`unknown state`**: target state isn't declared on the board. Use
  `list_issues` to discover the board, or run
  `iterion issue board show` from a shell.
- **`board has no terminal state`**: only fires on `close_issue` when
  no terminal state is configured. Pass `to:` explicitly or fix the
  board.
- **`required field <name> missing`**: the board declares a required
  custom field that wasn't provided. Add the field to
  `create_issue.fields` or unset the requirement.

## Sandbox runs (containerised bots)

When the workflow runs with `sandbox:` active, the host's iterion
binary isn't reachable inside the container. The runtime falls back
to an HTTP transport (`/api/v1/mcp/board` on the iterion server),
authenticated via an ephemeral `X-Iterion-Run` token. Both paths
expose the same tool set and the same error semantics.

## Quick sanity check

After your workflow run finishes, you can inspect the resulting
board from a shell:

```bash
iterion issue list --store-dir <workspace>/.iterion/dispatcher
iterion issue board show --store-dir <workspace>/.iterion/dispatcher
```

Or via the studio at `/board` if `iterion studio` is running on the
same workspace.
